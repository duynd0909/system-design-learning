# Migration Plan: first-start-questions.json + Per-Problem Component Palette

## Context

The current seed has 10 problems (Instagram, YouTube, etc.) with a progressive requirement system where each requirement *introduces new nodes* into the graph. The component palette is global (all 12 ComponentType records shown for every problem).

The new source-of-truth is `docs/first-start-questions.json` (12 problems) with a different structure:
- **One complete design graph per problem** — all nodes (revealed + blank) in a single `design` object
- **Blank nodes pre-defined** (`isBlank: true` + `hint`) rather than randomly masked
- **`component_options`** per problem — a curated subset of component types for the palette (6–12 options, includes distractors)
- **Requirements reference `node_ids`** from the design — they group existing nodes for scoring, not introduce new ones

**Goal:**
1. Replace the 10 current problems with 12 new ones from `first-start-questions.json`
2. Make the component palette per-problem (driven by `component_options`)
3. Cache component types in Redis (infrequently changed)
4. Clean up legacy masking/seeding logic no longer needed after migration

**Game flow stays progressive (accumulated):** Each requirement's canvas = nodes from reqs 1..N accumulated. Blank nodes for req N = blank nodes first appearing in req N's `node_ids`.

**Phase 2 (not this plan):** Add 56 problems from `docs/next-phase-questions.json`.

### Task 3 — Seed New Problems

#### 3a. Algorithm: Split Complete Design into Per-Requirement Nodes

The design has ONE flat `nodes` array. Requirements have `node_ids` referencing those nodes.
Nodes appearing in multiple requirements' `node_ids` belong to the **first** requirement they appear in.

```typescript
function splitDesignToRequirements(design, requirements) {
  const nodeMap = new Map(design.nodes.map(n => [n.id, n]));
  const assignedIds = new Set<string>();

  return sortBy(requirements, r => r.id /* original order */).map(req => {
    const inDesign = req.node_ids.filter(id => nodeMap.has(id));
    const newIds = inDesign.filter(id => !assignedIds.has(id));
    newIds.forEach(id => assignedIds.add(id));
    return { req, newNodeIds: newIds, accumulatedIds: new Set(assignedIds) };
  });
}
```

**Unknown node IDs** in `node_ids` (not in `design.nodes`) are skipped — they are solution-graph nodes from the source system not present in the problem graph.

#### 3b. Edge Assignment

An edge belongs to requirement N if **at least one endpoint is new in req N** AND **both endpoints are in the accumulated node set for req N**:

```typescript
function assignEdgesToRequirements(edges, perReqData) {
  return perReqData.map(({ newNodeIds, accumulatedIds }) => {
    const newSet = new Set(newNodeIds);
    return edges.filter(e =>
      accumulatedIds.has(e.source) &&
      accumulatedIds.has(e.target) &&
      (newSet.has(e.source) || newSet.has(e.target))
    );
  });
}
```

#### 3c. Node Type Mapping

| Source `data.componentType` | DB `node.type` | Notes |
|-----------------------------|----------------|-------|
| `DESKTOP`, `CLIENT` | `actor` | User-side actors, never blank |
| `BLANK` (`isBlank: true`) | `component` | Stored with inferred correct slug + hint |
| Everything else | `component` | Revealed component |

Node data:
```typescript

{ componentSlug: 'load-balancer', label: 'Load Balancer', hint?: 'Which component...' }
// actor node
{ label: 'Desktop Client' }
```

Blank nodes are stored as `component` with the correct answer slug. `buildAccumulatedGraph()` converts them to `blank` type at runtime. The `hint` survives masking.

async findAll(): Promise<ComponentType[]> {
  const cached = await this.redis.get('components:all');
